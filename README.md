# Template Admin with Login 2FA

A production-ready SaaS Admin template built with Next.js 16 (App Router), Tailwind CSS v4, and shadcn/ui. This boilerplate features a complete authentication flow (JWT + MFA), Role-Based Access Control (RBAC), and beautifully crafted UI components using TanStack Table and Recharts.

## Features

- **Next.js 16 App Router**: Leveraging the latest features of Next.js with React Server Components.
- **JavaScript Only**: Clean, readable JavaScript implementation without TypeScript overhead.
- **Authentication**: JWT-based auth via HTTP-only cookies with TOTP (Google Authenticator) 2FA support.
- **RBAC**: Fine-grained role-based access control with middleware route protection and UI components.
- **Modern UI**: Powered by Tailwind CSS v4 and highly customizable shadcn/ui components.
- **Charts & Tables**: Integrated Recharts for data visualization and TanStack Table for powerful data grids.
- **Mock Backend**: Includes an in-memory mock database and simulated API delays (500ms) for realistic development.

## Architecture details

### How JWT Works
Upon successful login (and MFA verification), the server signs a JWT using the `jose` library (which is Edge-runtime compatible). The token payload contains the user's `id`, `email`, `role`, and an array of `permissions`. This token is set as an **HTTP-only cookie** (`auth_token`), protecting it from XSS attacks. The Next.js middleware reads this cookie to protect routes, and API endpoints read it to authorize actions.

### How MFA (TOTP) Works
When a user registers, a unique TOTP secret is generated using `otpauth`. A QR code is presented to the user to scan with Google Authenticator or Authy. During login, after validating credentials, the user is redirected to the `/mfa` route. The user must provide the 6-digit code generated by their app, which the server verifies against their stored TOTP secret.

### How RBAC Works
Role-Based Access Control is enforced at three layers:
1. **Middleware Level**: `src/middleware.js` checks the requested pathname against a route-to-permission dictionary (`ROUTE_PERMISSIONS`). If the user's JWT payload lacks the required permission, they are redirected.
2. **API Level**: Endpoints like `GET /api/users` verify the token and check explicitly if the user has specific permissions (e.g., `hasPermission(user, 'VIEW_USERS')`).
3. **UI Level**: The `<Can permission="...">` component conditionally renders UI elements (like the "Add User" button or navigation links) only if the authenticated user has the specified permission.

## Deployment (Vercel)

This project is perfectly suited for deployment on Vercel as it relies entirely on Next.js features and Edge middleware.

1. Push your code to a GitHub repository.
2. Import the project into Vercel.
3. Vercel will automatically detect the Next.js framework.
4. Add environment variables if you transition away from the mock database.
5. Deploy.

> **Note**: Because this boilerplate uses an *in-memory* mock database (`src/mock/db.js`), data changes (like new users) will reset whenever the Node.js server restarts or when serverless functions spin down.

## Extending the Template

### Switching to a Real Backend
1. Delete the `src/mock/` folder and `src/app/api/` folder.
2. Update the `BASE_URL` in `src/lib/api.js` to point to your real backend API.
3. Update `src/features/auth/auth.service.js` and `src/features/users/users.service.js` to match your actual API contracts.
4. Ensure your real backend handles JWT generation and sets the HTTP-only cookie identically so Next.js middleware continues to function.

### Adding New Roles or Permissions
1. Open `src/mock/roles.js`.
2. Add your new permission to the `PERMISSIONS` object.
3. Add your new role to the `ROLES` object.
4. Update the `ROLE_PERMISSIONS` mapping to connect your new role to the desired permissions arrays.
5. In `src/middleware.js`, map any new routes to the required permissions.
